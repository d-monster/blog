mysql-prox搭建以及遇到的问题

  • proxy：uhost-ubuntu
  • mysql：udb

描述：因为udb在内网，不能通过外网直接访问，所以需要一个连接eip的uhost来做proxy，且uhost和MySQL须在同一内网中互相能ping通

搭建过程

  • 下载安装

    apt-get install mysql-proxy

  • 获取版本信息

    mysql-proxy -V


    ps：注意.lua文件

  • 配置

    vi /etc/mysql-proxy.cnf

输入：
    [mysql-proxy]            #标志
    admin-username=root     #admin用户名
    admin-password=123fsck    #admin密码
    admin-lua-script=/usr/lib64/mysql-proxy/lua/admin.lua    #lua位置，参见上面的版本信息
    daemon=true        # mysql-proxy以守护进程方式运行
    keepalive=true        #保持连接启动进程会有2个， 一号进程用来监视二号进程
    proxy-backend-addresses=10.6.X.XX  #目标地址，udb内网地址，默认端口3306
    log-file=/var/log/mysql-proxy.log
    log-level=debug

chmod 0660 /etc/mysql-proxy.cnf   //更改权限

  • 开启服务

    mysql-proxy --defaults-file=/etc/mysql-proxy.cnf

问题：

  1. uhost的ubuntu的apt源过旧，无法安装mysql-proxy

    1. 添加apt-get源： 

      vi /etc/apt/sources.list

      源可参考：国内外apt源

    2. 添加完之后: 

      apt-get update  //更新软件源

  2. 配置完成后依旧无法添加

    1. 在控制台登陆mysql，检查用户名，密码是否正确

    2. 在uhost上远程登陆mysql，检查udb的mysql当前用户是否允许远程登陆

    3. 检查当前主机网络是否连通uhost

    4. 检查uhost外网防火墙是否打开proxy配置的端口

centos上搭建shadowsocks的两种方法

这两个月刚好在XXX公司实习，给了我一个账号买主机实验用，于是利用工作之便搭建了一个shadowsocks的服务器。github上找了一下，大概发现两个可用的。

  • ps：主机为centosos6.5

方法一 python

这个方法搭建的服务器只能以进程运行，不能搭建服务。

  1. 安装pip

    yum install python-pip

  2. 使用pip安装shadowsocks

    pip install shadowsocks

  3. 配置Shdowsocks服务,并启动

    用vim新建 /etc/shadowsocks.json 文件, 并写入以下内容

    {
"server":"0.0.0.0",  //指定连接的ip（所有人都可连接就为0.0.0.0）
"server_port":443,    //指定服务端口
"local_address":"127.0.0.1",
"local_port":1080,
"password":"your-passwd",    //设置密码
"timeout":300,
"method":"aes-256-cfb",    //设置加密方式
"fast_open":false,
"workers":5        //启动的进程数
}

    还有多用户模式的配置我就不弄了，可自行百度，因为就我自己用，而且单用户也可以多方连接。

    启动:ssserver -c /etc/shadowsocks.json -d start 
关闭:ssserver -c /etc/shadowsocks.json -d stop

方法二 脚本搭建

这个方法可以启服务

  1. 下载安装

    cd /tmp
wget --no-check-certificate https://raw.githubusercontent.com/teddysun/shadowsocks_install/master/shadowsocks-libev.sh
chmod +x shadowsocks-libev.sh
./shadowsocks-libev.sh 2>&1 | tee shadowsocks-libev.log

  2. 按照提示步骤安装即可

    看到enjoy it！就说明完成了

  3. 查看是否安装完成

    ps -ef | grep ss-server | grep -v ps | grep -v grep

  4. 配置
    配置文件在 /etc/shadowsocks-libev/config.json

    {
   "server":["[::0]","0.0.0.0"], //ipv4/ipv6
"server_port":your_server_port,
"local_address":"127.0.0.1",
"local_port":1080,
"password":"your_password",
"timeout":600,
"method":"aes-256-cfb"
}

    基本相同，只不过支持ipv6

  5. 卸载

    cd /tmp  # 进入 shadowsocks-libev.sh 所在目录
./shadowsocks-libev.sh uninstall

  6. 启动

    service shadowsocks start   # 启动
service shadowsocks stop    # 停止
service shadowsocks restart # 重启

故障排查

  • 问题：访问谷歌出现505/500报错，不能翻

    因为我是在云主机上搭建的，云主机外网ip在pnat设备上，不再本机，有外设防火墙，所以要到控制台上设置一下规则，允许设置的端口走tcp流量

  • 指定端口时不建议用常用端口，容易被墙

  • 如果要在云主机上搭建，主机配置最低就行，但是eip最好带宽大一点，不然速度慢

CDN内容分发网络

CDN架构

最简单的CDN网络只需一台负责全局负载均衡的DNS和中心节点，边缘节点各节点一台Cache，即可运行。DNS支持根据用户源IP地址解析不同的IP，实现就近访问。为了保证高可用性等，CDN网管中心需要监控各节点的流量、健康状况等。一个节点的单台Cache承载数量不够时，才需要多台Cache，多台Cache同时工作时，才需要负载均衡器，使Cache群协同工作。

中心节点

中心节点包括CDN网管中心和全局负载均衡DNS重定向解析系统，负责整个CDN网络的分发及管理。

CDN网管中心

CDN网管中心是整个CDN能够正常运转的基础保证，它不仅能对整个CDN网络中的各个子系统和设备进行实时监控，对各种故障产生相应的告警，还可以实时监测到系统中总的流量和各节点的流量，并保存在系统数据库中，使网管人员能够方便地进行进一步分析。一套完善的网管系统，允许用户按需对系统配置进行修改。

全局负载均衡DNS

全局负载均衡DNS通过一组预先定义好的策略，将当时最接近用户的Cache节点地址提供给用户，使用户能够得到快速的服务。同时，它还与分布在各地的所有CDN节点保持持续通信，搜集各节点的通信状态，确保不会将用户的请求分发到不可用、或不健康的Cache节点上。

边缘节点

CDN边缘节点主要指异地分发节点，由负载均衡设备、高速缓存服务器两部分组成。

负载均衡设备

负载均衡设备负责每个节点中各个Cache的负载均衡，保证节点的工作效率；同时还负责收集节点与周围环境的信息，保持与全局负载均衡DNS的通信，实现整个系统的负载均衡。

##高速缓存服务器
高速缓存服务器（Cache）负责存储客户网站的大量信息，就像一个靠近用户的网站服务器一样响应本地用户的访问请求。通过全局负载均衡DNS的控制，用户的请求被透明地指向离他最近的节点，节点中Cache服务器就像网站的原始服务器一样，响应终端用户的请求。因其距离用户更近，故其响应时间才更快。

CDN四大关键技术

内容路由技术

CDN负载均衡系统实现CDN的内容路由功能。它的作用是将用户的请求导向整个CDN网络中的最佳节点。最佳节点的选定可以根据多种策略，例如距离最近、节点负载最轻等。负载均衡系统是整个CDN的核心。

  • 通常负载均衡可以分为两个层次:全局负载均衡(GSLB)和本地负载均衡(SLB)。

    • 全局负载均衡(GSLB)主要的目的是在整个网络范围内将用户的请求定向到最近的节点(或者区域)。

      对于全局负载均衡而言，为了执行就近性判断，通常可以采用两种方式，一种是静态的配置，例如根据静态的IP地址配置表进行IP地址到CDN节点的映射。另一种方式是动态的检测，例如实时地让CDN节点探测到目标IP的距离(可以采用RRT，Hops作为度量单位)，然后比较探测结果进行负载均衡。当然，静态和动态的方式也可以综合起来使用。

    • 本地负载均衡一般局限于一定的区域范围内，其目标是在特定的区域范围内寻找一台最适合的节点提供服务，因此，CDN节点的健康性、负载情况、支持的媒体格式等运行状态是本地负载均衡进行决策的主要依据。

      对于本地负载均衡而言，为了执行有效的决策，需要实时地获取Cache设备的运行状态。获取的方法一般有两种，一种是主动探测，一种是协议交互。主动探测针对SLB设备和Cache设备没有协议交互接口的情况，通过ping等命令主动发起探测，根据返回结果分析状态。另一种是协议交互，即SLB和Cache根据事先定义好的协议实时交换运行状态信息，以便进行负载均衡。比较而言，协议交互比探测方式要准确可靠，但是目前尚没有标准的协议，各厂家的实现一般仅是私有协议，互通比较困难。

  • 负载均衡可以通过多种方法实现，主要的方法包括DNS、应用层重定向、传输层重定向等等。

内容分发技术

内容分发包含从源到CDN边缘的Cache的过程。从实现上看，有两种主流的内容分发技术:PUSH和PULL.

  • push是一种主动分发技术

    通常，push有内容管理系统发起，将内容从内容源芬达到边缘cache节点。分发协议可采用http/ftp等

  • pull是一种被动分发技术

    PULL分发通常由用户请求驱动。当用户请求的内容在本地的边缘Cache上不存在(未命中)时，Cache启动PULL方法从内容源或者其他CDN节点实时获取内容。

内容存储技术

对于CDN系统而言，需要考虑两个方面的内容存储问题。一个是内容源的存储，一个是内容在Cache节点中的存储

  • 对于内容源的存储，由于内容的规模比较大(通常可以达到几个甚至几十个TB)，而且内容的吞吐量较大，因此，通常采用海量存储架构。如NAS和SON.
  • 对于在Cache节点中的存储，是Cache设计的一个关键问题。需要考虑的因素包括功能和性能两个方面:在功能上包括对各种内容格式的支持、对部分缓存的支持，在性能上包括支持的容量、多文件吞吐率、可靠性、稳定性。

内容管理技术

  • 本地内容索引。 本地内容管理首先依赖于对本地内容的了解。包括每个Cache设备上内容的名称、URL、更新时间、内容信息等。本地内容索引是实现基于内容感知的调度的关键。

  • 本地内容拷贝。通常，为了提高存储效率，同一个内容在一个CDN节点中仅存储一份，即仅存储在某个特定的Cache上。但是一旦对该内容的访问超过该Cache的服务提供能力，就需要在本地(而不是通过PUSL的方式)实现内容的分发。这样可以大大提高效率。

  • 本地内容访问状态信息收集。搜集各个Cache设备上各个内容访问的统计信息，Cache设备的可用服务提供能力及内容变化的情况。

可以看出，通过本地内容管理，可以将内容的管理从原来的Cache设备一级，提高到CDN节点一级，从而大大增加了CDN的可扩展性和综合能力。

网络安全

vpn

虚拟的专有网络，
可以对数据进行加密和完整性保护，也可以不加密，不保护

  1. vpn分类

    1. osi7层
      1. 应用层 ：app 开源vpn（openvpn）
      2. 会话层，表示层 ：没有vpn
      3. 传输层 ：ssl vpn（安全 socket（套接字/端口））
      4. 网络层 ：ipsec vpn（运用度很广）mpls vpn（2.5层）
      5. 数据链路层 :pptp（点到点隧道协议）,l2tp（二层隧道协议）—-逐渐被淘汰（用于保护操作系统数据传输过程）

    2. 应用

      1. L2L vpn；site to site vpn（局域网到局域网，站点到站点） ：

        ipsec vpn—->DM vpn(基于ipsec，为了取代它，思科私有)【ccie考试必考】

        get vpn （通用的 封装加密 隧道）对mpls vpn的安全加固（思科私有）

      2. remote vpn

        ssl vpn（保护邮件）

        pptp，l2tp ez-vpn（思科私有）—->anyconnect vpn（最新）能适配终端更多（ios/Android…）

    3. 明文数据挑战
      1. 对等体认证
      2. 完整性
      3. 加密
      4. 抵赖—-数字签名

    4. vpn组建

      1. key 密钥

        1. 认证的过程中 key 就是口令
        2. 计算过程中 key是工具，变量

        3. 加密算法key

          1. 对称key ：加密解密同一把key，优点：快，高效，加密后的数据以及不会膨胀太多

          2. 非对称key ：

            1. 公key

            2. 私key

              拿到公私钥互相算不出对方，但可以互相解密

            3. 一般情况下，加密公私钥如何分配

              公钥互相交换，公钥加密数据，私钥解密

            4. 数字签名

          对称密钥加密算法保护数据，非对称密钥加密算法保护过程

    5. 对称，非对称算法

      1. 对称的 ：des；3des；aes
      2. 非对称 ：dsa；rsa

    6. 数据完整性保护算法

      1. md5（消息摘要算法）算完128bit（描述一个算法的强壮程度是bit数）
      2. sha（算完256bit，要去代md5）

    7. key的产生

      dh算法 ，两个随机数算出相同的数，且不可逆（魔方算法）

    8. 启源认证/对等体认证

      1. 预共享口令认证
      2. 数字证书 rsa/signature
      3. 加密随机数（淘汰，傻逼）

    9. lsakmp与ike2个阶段

      1. lsakmp互联网安全关联密钥管理协议
        1. sa安全关联—-连接和过程已经被保护了

      2. ike互联网密钥交换（被包含在isakmp）

        a----------------b
1. ike阶段一
    1. （1.2）互发vpn保护策略（方式一样）
    2. （3.4）互发认证消息
    3. （5.6）互发保护vpn的key
2. ike阶段二
    1. （7.8.9）利用阶段一的key算出保护数据的key，确认加密用户数据的算法和完整性的算法

    10. vpn头部

      1. 隧道模式 tunnel mode
      2. 传输模式 transport mode
      3. ah，esp
        1. ah认证头：加密数据只保证认证和完整性
        2. ESP：封装安全静荷（静荷就是中心数据）：加密数据保证认证和完整性

ipsec vpn的配置

  1. 确认路由器ios支持ipsec vpn

    show version //查看版本
K9的含义代表支持128bit以上的加密算法

  2. 完成ipsec阶段一保护策略的设置

    crypto isakmp policy <策略编号>
authentication pre-share //开启预共享口令认证
encrption <des/3des/aes> //设置加密vpn数据流量方式
hash <md5/sha> //设置vpn流量完整性保护方式
group <1/2/5>  //设置dh组算法级别

  3. 设置对等体地址，并设置预共享口令

    crypto iskmp key <口令> address <对端vpn网关公网地址>

  4. 设置ike阶段二 ，用户数据保护方式和数据封装头部方式和vpn方式

mode <tunnel/transport>
  5. 编写扩展acl，利用acl匹配需要被vpn保护的流量

  6. 配置加密映射，将对等体，感兴趣流和阶段二用户保护方式结合

    crypto map 只对接口发送的流量生效

    show crypto engine connection active  //查看vpn是否机密解密流量

ipsec vpn与nat同时存在的问题

nat会先于vpn抓取到流量，所以nat的转换工作可能会导致vpn无法识别该被加密的流量

  1. nat bypass ：nat绕行

    改acl，使通过vpn到达对端的流量不走nat

ipsec vpn实施方式

  1. 隧道ipsec vpn

    先建立gre隧道，利用隧道完成路由可达，再利用ipsec vpn对隧道保护

  2. 设备出口直接使用ipsec vpn

    设备出口流量会被ipsec vpn识别，判断是否需要加密保护

ipsec vpn的缺陷

  1. 对端设备必须公网地址固定
  2. 如果需要全互联IPSEC VPN保护，配置特别复杂
  3. 无法实现分支之间无需IPSEC VPN配置就能建立IPSEC VPN连接

ISIS

IS-IS

  1. 中间系统到中间系统（属于igps内部网关协议集），link state链路状态协议（如ospf）

  2. 发展历程

    • 有链接 cons ip网络标识 -> tcp -> 路由协议
    • 无连接 clns clnp无连接的网络协议，是clns网络标识 -> clns无连接的网络服务机制，后改名为udp -> 路由协议（IS-IS/集成的IS-IS/双栈的IS-IS）

  3. clnp标识

    1. NET网络实体标识（类似ip+mac）
    2. NASP网络服务接入点（类似ip+mac+port_num端口号）

    3. CLNP标识格式

      AFI.0000.sys-id.00
地址格式授权组织.域标识（AS）区域标识.系统id（mac地址）.端口号

      最后一部分若为00则为NET，若为非00则为NSAP，
      IS-IS启用的时候必须要有一个NET地址

  4. IS-IS术语

    1. ES终端设备
    2. IS：router
    3. clnp无连接网络协议
    4. clns无连接网络服务
    5. dis指定中间系统（类似DR）
    6. IS-IS中路由器的类型（IS-IS router type）
      1. 所有启用IS-IS的路由器默认类型为L1/L2（类似ABR）
      2. L1:区域内路由器
        3 .L2:骨干区域路由器

    7. IS-IS neighbor type

      1. L1/L2
      2. L2

      3. L1

        相同区域id            不同区域id          
L1-L1-L1/L2            L1--/--L1/L2
L1-L1-L1            L1--/--L1
L1--/--L2            L1--/--L2
//L1设备只能跟相同区域内带L1角色的设备形成L1邻居关系即区域内邻居关系
L1/L2--L1或L2--L1/L2        L1/L2--L2--L1/L2
//L1是区域内邻居关系，L2首先是区域间邻居关系，但也可以是区域内邻居关系
L1/L2--L2--L2        L1/L2--L2--L2

    8. IS-IS特性

      1. IS-IS中dis特点
        ospf中dr作用 1.减少不必要的邻接关系；
        2.统筹发送更新过程
        1. dis的选举 先比较优先级（默认是64，最大127，大则优）
          再比较sys-id
        2. 没有bdis，且dis可以虽已被替代
        3. 所有非dis之间都可以形成正常邻接关系
      2. IS-IS中区域的特点 IS-IS也是层次化的设计，骨干层，常规层，同样也有骨干区域和常规区域
        1. 不要求骨干区域编号一定是0
        2. 不要求所有常规区域和骨干区域连接
      3. IS-IS区域的边界是在路由器，ospf是在接口，ospf中更改区域号必定终端网络，IS-IS中可以先启好区域在删除原有区域
      4. ospf中拥有特殊区域stub nssa， IS-IS区域内设备只要路由器角色改为L1，他所连接的L1/L2设备自动给他一条缺省，并且不会将区域间路由和外部路由放入区域内

    9. is-is路劲度量值计算，计算方法和ospf一样，cost ，默认情况下所有接口都被赋予10的开销

    10. IS-IS的消息和消息类型

      1. IIH：IS-IS hello 分为L1和L2
      2. LSP： link state pdu 分为L1和L2

      IS-IS数据库也分L1和L2

    11. IS-IS的配置

      • 步骤1：启用Is-IS平设定NET地址

        router isis
net 49.0.0000.0000.00000001.00
log-adjacency-changes//让控制台以日志的形式提醒IS-IS邻接关系的变化

      • 步骤2：接口加入IS-IS

        interface <intf-id>
ip router isis

    12. IS-IS的查看

      1. 查看路由表中IsIS的条目

        show ip route isis
: i L1 区域内路由
    i L2 区域间路由/外部路由

        IS-IS管理距离全是115

      2. 查看邻接关系是否建立，以及邻接关系的类型

        show clns neighbor

      3. 查看dis (circuit id ==dis)

        show isis neighbor

    13. 改路由器的类型

      router isis 
is-type level-1 //将路由器类型改为L1

    14. 修改优先级和开销

      interface x 
isis priority 127   //优先级

isis metric x        //开销

    15. IS-IS重分发

      外部网络进入IS-IS默认是L2类型，默认开销是10

      router isis
redistribute <协议> metric <指定外部路由初始度量值> <level-1/level-2/level-1-2> //指定外部路由是什么类型进入isis

redistribute ospf pid metric-type <external ospf 的外部路由/internal ospf 的区域内或区域间路由>

      路由器角色为L1，那么重分发不允许做在这样的设备上，如果L1设备需要配置上重分发，那么务必在后加上level-1

    16. IS-IS的手动汇总

      汇总配置在L1/L2角色上，汇总后路由默认类型为L2，开销为10

      L1设备上默认无法对本地isisL1路由直接汇总，本地L1网络以外部路由形式进入，再对L1路由汇总

      router isis
summary-address <汇总后的网络> <汇总后的子网掩码> level-1/level-2

    17. IS-IS认证

      1. 链路认证

        该认证是通过iih完成的，所以认证不通过影响邻接关系建立

        interface x
isis password xxx

      2. 区域认证

        该认证是利用lsp完成的，认证不通过只影响陆游的学习，不影响邻接关系

        router isis
area-password xxxx

      3. 域认证

        router isis
domanin-password xxxx

        该认证作用于区域间，如果不通过不影响邻接关系，只影响路由学习

    18. IS-IS路由泄露

      L1/L2默认不允许L2数据库信息进入L1数据库，变成更新发送

      路由泄露的配置流程

      步骤1：L1/L2设备上编写acl，匹配泄漏的目标路由

      步骤2：L1/L2设备上编写router-map引用acl

      步骤3：L1/L2设备上做isis到isis的重分发，并且指定是利用router-map完成L2数据条目到L1数据库条目的泄漏 

      redistribute isis ip level-2 into level-1 route-map 2T1

    19. IS-IS的不同区域迁移

      IS-IS允许一个设备属于不同的区域

      router isis
net ...........(此条命令可被复敲，不会覆盖)

多播

多播

  1. 数据传播方式

    1. 单播 ： unicast

      1对1，源和目的都是已知的

    2. 广播 ： broadcast

      1对all 源已知，目的通常未知

    3. 多播 ： mutiplecast（iphone iptv ipmeet）

      1对many或者many对1 源已知，目的通常未知

  2. 设备处理

    1. router

      1. 单播
      2. 广播 路由器默认阻止广播流穿越，除非有特殊的进程或协议
      3. 多播 默认不允许传递，除非有特殊的进程或协议

    2. switch

      1. 单播帧
        1. 已知目的mac单播帧 ： 定向转发
        2. 未知目的mac单播帧 ： 泛洪
      2. 广播帧 泛洪
      3. 多播帧 泛洪

  3. 多播的术语

    1. 多播源（可能是多个）
    2. 多播的成员 组成员
    3. 组group
    4. 树 路径（router提供）
    5. 源树：树根基于源
    6. 共享树：双层，下游设备基于rp算路径，rp基于源算路径
    7. 集合点rp

  4. 多播图

    1. igmp 互联网组管理协议

      igmp帮助路由器发现直连子网内的组成员

      igm帮助终端主动申告是不是组成员

      1. igmpv1

        1. report 报告消息，源于pc 毎60s发一次

          作用： 回应，申告

        2. request 查询消息 源于路由器

          查询回应等待时间10s

          老化时间180s

        3. 重大缺点

          1. 显性加入，隐性离开
          2. 多查询者
          3. 多个转发者

      2. igmpv2

        1. request
        2. report
        3. leave 来自pc 显性离开
        4. sq 特定查询消息 等待时间3s—-report

        5. 查询者竞选机制

          源ip越小越优

  5. 多播的ip和多播的mac

    1. 多播的目的ip

      224.xxx.0-239.xxx

      多播ip标识的不是数据接收设备，是服务或应用

      224.0.0.1 所有pc接收
224.0.0.2 所有路由器侦听
224.0.0.9 代表ripv2
224.0.0.10 代表eigrp

    2. 多播的mac

      01005E.后6个16进制书利用多播ip进行计算

      ！！注意 ： 多播mac地址可能因为多播ip的后23bit一致，而重复

  6. 多播路由协议

    主要负责计算多播路径（树），同时维护

    1. 多播路由协议的种类

      1. dvmrp：距离矢量多播路由协议：淘汰
      2. mospf：多播的ospf ：淘汰
      3. cbt：基于核心的树：淘汰
      4. pim：协议独立的多播路由协议：主流
      5. M-BGP：多播的bgp
      6. msdp：快于多播时使用，多播源发现协议

    2. PIM

      as内负责多播树构建的协议

      1. pim的模式

        1. 密集模式：路径都是源树

          子网内多播组成员占比重

          pim dense mode 使用的是igmpv1

        2. 稀疏模式：路径都是共享树

          pim sparse mode 使用的是igmpv2

        3. 注意

          pim的密集模式现在用的也是igmpv2

        4. pim的密集模式详解

          1. 密集模式的消息 扩散–修剪

            1. hello消息：用于建立pim邻居关系
            2. jion消息：当前路由器发给上游设备，告诉上游路径将直连路径加入多播树
            3. prune消息：修剪消息，当前路由器发给上游设备，告知上游设备将直连网络从树中移开
            4. graft消息：接枝消息，上游设备发送给上游，将之前移开的直连网络请求再加进去
            5. graft-ack：接枝确认消息，上游发给下游设备，告知路径已经接回来
            6. assert：禁言消息

          2. pim的密集模式的禁言消息

            pim assert消息实际上是竞选转发者所用

            assert消息里携带3个参数

            1. 转发多播流设备使用的底层协议管理距离 小优
            2. 转发多播流设备蒂法多播源的路径开销 小优
            3. 转发多播流接口IP地址 ip地址大优

          3. pim密集模式中的dr（指定路由器）

            pim的密集模式使用的是ibmpv1，pim的密集模式会在连接的子网一段竞选bdr，有dr最终负责查训工作。

            dr竞选过程：优先级和接口ip，优先级越大越优，接口ip越大越优

            pim的密集模式现在使用的是igmpv2，igmpv2小优，dr是ip大优，两者冲突，最终igmpv2决定一切

        5. pim稀疏模式：

          1. pim稀疏模式消息

            1. hello
            2. jion
            3. prune
            4. graft
            5. graft-ack
            6. assert

            7. register 注册消息 ：第一跳路由器发送给rp

              注册消息实际上是第一跳路由器将接收的多播流二次封装成单播流量，以抵达rp的最短路径发给rp，触发rp计算抵达多播源的最短路径计算

            8. register-stop 注册停止消息 rp反发给第一跳路由器

              rp以源为根，计算完毕最短路径后，利用注册停止信息，告诉第一跳路由器，不需要二次封装发送数据，路径完成可以多播形式发送

              多播源主动注册，下游主动加入

          2. pim sparse mode中dr的作用

            下游设备dr负责发送加入消息，

            多个第一跳设备dr负责发送注册消息

          3. 共享树切换成源树

            启用pimsm的设备接收超过4个字节的多播流之后，就会尝试以原为根计算最短路径，用该路径代替之前的共享树

            pimsm最终稳定之后实际上能看到的都是源树

            ！！注意：既然最终都切换到源树，那干嘛还要rp共享树呢 ： rp就是无扩散多播流得到多播路径的精髓，有了rp，有了共享树，有了更高效计算路径的方式，但未必路径最优，才需要切到源树

          4. 多播防环机制

            rpf revense path forward：默认开启帮助多播避环的机制，接收多播流的接口必须是单播路径上抵达多播源的出后，接口才可以接收，流量接收后，才允许被发送

          5. 稀疏模式的rp由来

            1. 人为手动指定， 维护不易，没有必要的冗余备份

              ip pim rp-address <xxxx>

            2. 动态竞选

              1. auto-rp：思科私有

                c-rp：：候选rp，所有期望成为rp的角色

                rp-agent：rp代理，获取c-rp信息，裁决出结果，共享给所有pim路由器

                rp-agent实干梨园手动指定，没有备份

                c-rp的竞选参数是c-rp的id，这个id通常是loopback接口IP管理员指定c-rp id越大越有可能成为rp

                1. auto-rp使用地址

                  224.0.1.39 RP-AGENT侦听，C-RP将信息发向这里。
                  224.0.1.40 所有启用PIM的路由器侦听，RP-AGENT将结果发向这里。

                  ！！注意 ，RP-AGENGT消息的传播范围TTL：默认是1（必须改范围最大255），路由器对待RP-AGENT消息的方式：接收后并不继续传递。

                2. auto-rp listen

                  路由器开启该功能，路由器不光接收RP-AGENT，还会在TTL允许的情况下继续传递。

                3. rp-agent的配置

                  Ip pim send-rp-discovery loopback 0 scope 255
Ip pim send-rp-an lo 0 sc 255（C-RP配置C-RP）
ip pim autorp listener（路由侦听）

              2. bsr boot strap router ：自荐路由器

                c-rp

                c-bsr：候选bsr，负责收集c-rp信息，但是不裁决，当c-rp信息集发送给所有pim路由器，由他们自己完成裁决

                bsr竞选机制：先比较优先级（大优），再比较bsr id（大优）

                rp竞选机制：优先级（小优），hash长度（越长越好，但是现在不用了），c-rp id（大优）

                ip pim bsr-candidate lo 0  0(hsah) 0（优先级）
show ip pim bsr router  //看谁是bsr
ip pim rp-candicate lo0 priority 1  //设置c-rp
interface 
ip pim dr-proiority xxx//改dr优先级

              3. 注意

                静态RP使用的LOOPBACK接口不需要参加PIM，AUTO-RP，BSR,作为ID的LOOPBACK接口必须参与PIM。

          6. pim混合模式：sparse-dense-mode 混合模式

            混合模式下，稀疏模式先工作，稀疏模式崩溃后，密集模式工作

            ！！补充：官方提供了命令，可以不允许混合模式下，稀疏退回密集

  7. 多播路由表的查看

    show ip mroute //查看多播路由表  
show ip igmp group //查看组成员
show ip igmp interface <intf-id>  //查看接口的igmp参数
show ip pim neighbor //查看pim邻居
show ip rpf <真实源ip> //查看rpf 
clear ip mroute *  //清除mroute
show ip pim rp //查看rp是否跟组关联了
show ip pim rp mapping //查看当前设备是否有rp信息
ip pim spt-threshold infinity //使稀疏模式不切回密集模式

    多播路由表条目几乎都是流量触发而来

    每个条目的老化时间是3分钟

    1. 多播路由表中的副条目和子条目

      1. 多播路由表的父条目不负责转发数据，子条目才负责转发数据

        （源信息，组信息）老化计时器
incoming interface （上游接口，接收多播流的接口）  状态（forward/prune）/模式（dense/sparse）
outgoing interface （下游接口，转发多播流量接口）  状态/模式

      2. 如何区分父条目，子条目

        通常父条目源是* 上游接口为空，子条目是真正的多播源，上下游都有详细的信息

        路由器收到report消息，就会针对特定组创建父条目，子条目基本都是真是的多播流创建

  8. 多播配置流程

    1. 将路由器模拟成多播组成员

      ip igmp jion-group <多播地址>

      该命令可以变化参与，然后反复的录入，让一个接口十多个不同组的成员

    2. 路由器开启多播路由功能

      ip multicast-routing

    3. 接口下启用pim

      ip pim dense-mode
ip pim sparse-mode
ip pim sparse-dense-mode

    4. 那些接口参与pim

      1. 连接子网成员的接口必须参与（参与之后igmp就在路由器开启了）

      2. 连接多播流的接口必须参与

        因为接口想要成功的接收多播流，必须满足两个条件：第一：参与pim，第二：通过rpf检测

      3. 最终需要传递多播流的路径两端接口必须参与

跨域多播

  1. 跨域多播遇到的挑战，
    1. 靠近源的设备，收到注册消息，计算出抵达源的spt（最短路径树）但是得不到后半段路径消息
    2. 靠近组成员的设备，收到join后，只能算出后半段路径，缺乏抵达源的路径信息

  2. msdp：多播源发现协议

    ip msdp peer xxxx connect-source lo 100
show ip msdp summ    //看msdp邻居关系
show ip msdp sa-cache //看有没有sa消息

    msdp配置在不同域的rp上，msdp会建立基于tcp的邻接关系，靠近源一侧的msdp收到注册消息后，会利用msdp sa消息，告诉靠近组成员一侧的rp，谁是源

    ！！注意：跨域多播中还需要考虑rpf的问题，接收多播流的接口必须通过ref检测，否则不接受转发多播流

  3. m-bgp

    m-bgp是帮助参与多播的设备通过rpf检测的手段

    ！！rpf的作用

    rpf会避免多播流一环路方式传递

    rpf也会限制接口接收多播流

    源网络可以通过传递的协议学习，比如： rip 。。。。，还可以通过m-bgp学习。

    m-bgp学习的网络进入m-bgp转发表，但不进入路由表

  4. m-bgp的配置

    router bgp
    nei remote xx
    address ipv4 multicast
    nei xxxx act

    show ip bgp ipv4 muliticast //m-bgp转发表
    cle ip bgp ipv4 1 soft out

  1. anycast 范播

    完全基于msdp，多播中利用anycast，可以提升rp切换的效率

  2. IGMP SNOOPING（嗅探） 和CGMP (思科组管理协议)

    cgmp和igmp snooping 是运行在路由器和交换机上的，将多播流发送给该发送的设备

    switch ： ip cgmp enable
router ： ip cgmp emable
